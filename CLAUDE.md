# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Product overview

This is an all-in-one project management app, aiming to cover what ClickUp, Jira, Slack, Notion, and a standalone LLM chat client each do separately, inside one project workspace. The pitch: instead of Jira for tasks, a separate chat app for the team, ChatGPT/Gemini in another tab, and Notion for notes, everything lives in the same project.

Per-project modules (see the sidebar nav built in [`Sidebar.tsx`](resources/js/components/layouts/Sidebar.tsx)): Board (Kanban), Chat, LLM chat, Notes, Calendar, Reminders, Pomodoro, and a Dashboard/Timeline view. Board, Chat, LLM chat, and Notes are built; the rest are planned at various stages, from nav-only placeholders to not started. Detailed behavior for each module (calendar conflict handling, presence visibility, etc.) will be documented here as those features get built out.

## Stack

Laravel 13 + Inertia.js (React 19, no SSR) + TypeScript + Tailwind v4. MySQL is the primary DB; MongoDB (`mongodb/laravel-mongodb`) is used for the LLM chat feature. Laravel Reverb (websockets) powers real-time chat. Google Gemini (`google-gemini-php/laravel`) powers the LLM chat feature. Route helpers are generated by Laravel Wayfinder.

## Commands

Run these in 4 separate terminals for full local dev (chat and LLM chat need the queue and Reverb):
```
php artisan serve
npm run dev
php artisan reverb:start
php artisan queue:listen --tries=1
```

Or run `composer run dev`, which starts the server, queue listener, and vite together (Reverb still needs its own terminal).

Frontend:
- `npm run lint` / `npm run lint:check`: ESLint (fix / check-only)
- `npm run format` / `npm run format:check`: Prettier over `resources/`
- `npm run types:check`: `tsc --noEmit`
- `npm run build`: production build (`build:ssr` also builds the SSR bundle, though SSR is disabled at runtime via `INERTIA_SSR_ENABLED=false`)

Backend:
- `composer lint` / `composer lint:check`: Pint (fix / check-only), runs `--parallel`
- `composer test`: clears config cache, runs `lint:check`, then `php artisan test`
- `php artisan test --filter=TestName`: run a single test
- `composer ci:check`: the full CI gate, `lint:check` + `format:check` + `types:check` + `test`. Run this before considering a change done.

Wayfinder route/action files under `resources/js/actions/`, `resources/js/routes/`, `resources/js/wayfinder/` are generated (the Vite plugin regenerates them on build/dev). Don't hand-edit them; they're also excluded from ESLint.

## Architecture

### Multi-account routing (`/u/{accountIndex}/...`)

Every authenticated route is nested under `Route::prefix('u/{accountIndex}')` (see [routes/web.php](routes/web.php)). This isn't just a URL quirk. It's how the app supports several logged-in accounts in one browser session, switchable without logging out.

- [`SetAccountRouteDefaults`](app/Http/Middleware/SetAccountRouteDefaults.php) runs on every request: it resolves which account the `{accountIndex}` (or the session, for legacy non-indexed routes) refers to, overrides the auth guard's user for that request via `Auth::setUser()` plus a custom user resolver, and stores the resolved index on `$request->attributes` as `account.index`. It also calls `URL::defaults(['accountIndex' => ...])` so Laravel route helpers don't need the param passed explicitly server-side.
- Account membership itself lives in `AccountSessionService` (a session-backed list of `{user_id, ...}` entries per browser session).
- [`HandleInertiaRequests`](app/Http/Middleware/HandleInertiaRequests.php) shares `account: { index, baseUrl }` and `accountsList` (all signed-in accounts, for the account switcher UI) with every Inertia page.
- On the frontend, **any route helper call (Wayfinder-generated or hand-written) must be passed the current `account.index`.** Read it from `usePage().props.account.index`, or via the `useProject()` / `useOptionalProject()` hooks, which already expose `accountIndex`. [`lib/accountRoutes.ts`](resources/js/lib/accountRoutes.ts) has plain-string helpers (`accountPath`, `projectPath`) for cases that don't go through Wayfinder.

### Project-scoped requests

Routes under `/u/{accountIndex}/p/{project:project_slug}/...` are additionally guarded by `project.member` (and `project.role:OWNER,ADMIN` / `project.role:OWNER` for privileged actions). See [`EnsureProjectMember`](app/Http/Middleware/EnsureProjectMember.php) and [`EnsureProjectRole`](app/Http/Middleware/EnsureProjectRole.php). When a route binds `{project:project_slug}`, `HandleInertiaRequests` auto-shares `project`, `projectRole`, `projectNavigation`, and `projectShare`, so pages under a project never need to declare these as props. They come through Inertia's global `sharedPageProps` (typed in [types/global.d.ts](resources/js/types/global.d.ts)) and are read via `useProject()` (throws outside a project route) or `useOptionalProject()` (nullable, for layout/header components rendered both in and out of a project context) in [hooks/useProject.ts](resources/js/hooks/useProject.ts).

Old un-indexed URLs (`/projects`, `/p/{slug}/...`) still resolve. They redirect to the account-indexed equivalent using the active session account, which keeps old links working from before multi-account existed.

### Frontend structure (`resources/js/`)

- `pages/`: one file per Inertia page (route target), mirroring the URL structure (`pages/projects/workspace.tsx`, `pages/auth/login.tsx`, etc). Pages compose layouts and components; they hold page-level state, not reusable logic.
- `layouts/`: shared page chrome (`AppLayout`, `AuthLayout`) wrapping page content.
- `components/`: grouped by feature/domain (`kanban/`, `chat/`, `llm-chat/`, `projects/`, `layouts/`), plus `shared/` for generic cross-feature UI (`Button`, `Checkbox`, `DatePicker`, `OtpInput`, etc). Feature folders with several related exports use a barrel `index.ts` (see `components/kanban/index.ts`), and the comments there point to where the corresponding types and utils live.
- `hooks/`: one hook (or a small related pair, like `useProject`/`useOptionalProject`) per file, named `use*.ts`.
- `types/`: one file per domain (`project.ts`, `kanban.ts`, `chat.ts`, `llm-chat.ts`, `auth.ts`), plus `global.d.ts`, which augments Inertia's `sharedPageProps` so every page gets typed access to shared props without a per-page interface.
- `lib/` vs `utils/`: both hold plain functions with no React dependency. In practice `lib/` is used for cross-cutting concerns (`utils.ts`'s `cn()` classname helper, `accountRoutes.ts`, `projectAvatar.ts`, `passwordStrength.ts`, `projectSlug.ts`) and `utils/` for feature-specific helpers colocated conceptually with a domain (`kanban.ts`, `chat.ts`, `attachmentStorage.ts`). When adding a helper, match whichever of the two the sibling logic already lives in rather than introducing a third location.
- `actions/`, `routes/`, `wayfinder/`: generated by Laravel Wayfinder. Import route/action helpers from these (e.g. `import projects from '@/routes/projects'`) instead of hardcoding URL strings.
- Path alias: `@/*` points to `resources/js/*`, `@public/*` points to `public/*` (see [vite.config.ts](vite.config.ts)).

### Internationalization (`resources/js/i18n/`)

The frontend has first-class i18n. All feature pages and user-facing feature UI should have matching translation configuration under `resources/js/i18n/locales`.

- Use `useTranslation()` from [hooks/useTranslation.ts](resources/js/hooks/useTranslation.ts) for user-facing copy in pages and components.
- Locale files live under `resources/js/i18n/locales/en/` and `resources/js/i18n/locales/id/`, grouped by feature/page namespace (`kanban.ts`, `chat.ts`, `notes.ts`, etc.).
- English locale files define the source shape. Indonesian files should import `typeof en` from the matching English file so TypeScript catches missing or extra keys.
- When adding a new feature page or user-facing feature area, add/update both the English and Indonesian locale files and wire any new namespace into [i18n/dictionary.ts](resources/js/i18n/dictionary.ts).
- Avoid hardcoded user-facing strings in React markup, buttons, form labels, empty states, alerts, modals, page titles, aria labels, and validation messages. Put them in the appropriate locale namespace instead.

### Icons

There's no icon component library. Icons are raw `.svg` files under `public/icons/...`, imported as React components via `vite-plugin-svgr` through the `@public` alias:
```ts
import AddIcon from '@public/icons/small/plus.svg';
```

### Styling (Tailwind v4)

Tailwind config is still the classic JS form ([tailwind.config.js](tailwind.config.js)), loaded into the v4 CSS pipeline via `@config` in [resources/css/app.css](resources/css/app.css), rather than the newer `@theme` inline-CSS-only approach. A small `@theme` block in `app.css` also defines a couple of raw CSS custom properties (`--header-height`, `--font-sans`) alongside the JS config.

- **Font sizes are named semantic tokens, not raw pixel utilities.** Body scale: `micro` (10px), `xsmall` (12px), `small`, `normal`, `medium`, `large` (20px). Heading scale: `h1`–`h6`, each bundling its own line-height and weight. Never use arbitrary values like `text-[13px]`. If the size you need doesn't exist, add a token to `tailwind.config.js` instead of reaching for an arbitrary value.
- **Color palette** is split into `dark.*` / `light.*` surface and text tokens, `accent.*` (10 named hues, each with a `DEFAULT` and `-light` variant, used for project/user avatar colors), and `status.*` (info/success/warning/error). Dark mode is the default/base palette. Light mode is implemented by overriding the `--color-dark-*` CSS variables under an `html.light-mode` selector in `app.css`, rather than a Tailwind `dark:` variant. Tailwind v4 exposes every config color as a `--color-{name}` custom property, so components just use `bg-dark-surface-1`, `text-dark-primary`, etc. and get themed automatically.
- **Dynamic/user-generated colors caveat**: values like a project or account avatar color are stored as plain strings (e.g. `"accent-blue"`) and can't be turned into a Tailwind class at runtime (`` `bg-${color}` ``), because Tailwind's compiler only picks up class names it can see statically. For these cases, use the hex lookup in [lib/projectAvatar.ts](resources/js/lib/projectAvatar.ts) (`AVATAR_COLOR_HEX` / `avatarHex()`) and apply the color via inline `style`, not a generated class name.
- Classnames are merged with `cn()` from [lib/utils.ts](resources/js/lib/utils.ts) (`clsx` + `tailwind-merge`). Prettier's `prettier-plugin-tailwindcss` is configured to sort classes inside `clsx`, `cn`, and `cva` calls.
- 2-space indentation in `.tsx`/`.ts`/`.css` (see `.editorconfig` / `.prettierrc`), single quotes, semicolons.

### Code style conventions (enforced by ESLint/Pint, not just habit)

- Components and hooks are arrow-function expressions (`const Foo = () => {}`), not `function Foo() {}`. This is enforced by `func-style`/`prefer-arrow-callback`.
- `import type { X }` is required for type-only imports (`consistent-type-imports`), and import groups (builtin/external/internal/parent/sibling/index) must be alphabetized.
- PHP is formatted with Pint (`composer lint`), config in [pint.json](pint.json).

### Data model notes

- `Project` uses a non-incrementing custom primary key, `project_id` (not the default auto-increment `id`), and slugs are generated/uniqued via `Project::generateUniqueSlug()`. Don't assume `$project->id` exists, and don't assume slugs are user-supplied verbatim.
- Project membership roles are the `ProjectRole` enum (`OWNER`, `ADMIN`, `MEMBER`, `VIEWER`), stored as the string value on the `project_user` pivot. `OWNER` is intentionally excluded from `assignableValues()`: a project always keeps exactly one owner, so role-management UI/endpoints must not offer reassigning ownership through the generic member-role update path.

## Engineering standards

These apply to any new code or change in this repo, not just the areas above.

- **Gate every change on `composer ci:check`** (backend: Pint + Pest) and, for frontend changes, `npm run lint:check && npm run format:check && npm run types:check`. A change isn't done until these pass. Don't hand-fix formatting/lint issues that the tools already fix automatically (`npm run lint`, `npm run format`, `composer lint`).
- **Split components by responsibility, following the existing `components/<feature>/` + `shared/` layout** (see Frontend structure above). If a page or component accumulates unrelated state or markup, extract that part into its own component/file rather than letting one file keep growing. Don't introduce a new top-level folder for this; fit into `pages/`, `layouts/`, `components/<feature>/`, `hooks/`, `lib/`/`utils/` as documented above.
- **Avoid N+1 queries.** Eloquent relations must be eager-loaded (`with()`/`load()`) before being accessed in a loop or a resource/map callback. This codebase already does that in places like `HandleInertiaRequests::resolveProjectShare()` (`$project->members()->select(...)->get()` then mapped once, invitations similarly scoped/limited up front). Prefer `select()`-ing only the columns a query needs, as in that same method, over pulling full rows when only a few fields are used.
- **Push side effects that don't block the response onto the queue** (`QUEUE_CONNECTION=database` locally) instead of doing them inline in a controller/request cycle. This is why `queue:listen` is one of the 4 dev processes.
- **Every feature needs both a Pest feature test and, where there's non-trivial standalone logic (services, enums, helpers in `lib/`/`utils/`, policies), a unit test.** Feature tests live in `tests/Feature/` and drive the app through routes/middleware like a real request (see [`MultiAccountSessionTest`](tests/Feature/MultiAccountSessionTest.php) for the `it(...)`/`expect(...)` Pest style, `RefreshDatabase`, and asserting via `assertRedirect`/session state). Unit tests live in `tests/Unit/` and exercise a single class/function in isolation, without touching the database or HTTP layer. When adding a controller action or Inertia page, add or extend the matching feature test in the same PR rather than leaving coverage for later.
- **Keep the file structure grouped by feature/domain, on both sides of the stack, and never dump a new file at the root of a shared folder just because it's convenient.** The backend already does this: `app/Http/Controllers/{Auth,Chat,Kanban}/`, `app/Http/Requests/Chat/`, `app/Models/LlmChat/`. A new feature gets its own subfolder the same way once it has more than one class, rather than piling flat files into `Controllers/` or `Models/`. On the frontend this is the `components/<feature>/`, `routes/<feature>/`, `types/<feature>.ts` grouping described above. Before adding a file, check whether a folder for that feature/domain already exists and put it there; only create a new one when the feature genuinely doesn't fit an existing grouping.
- **Always validate input in the controller before it touches a model or service.** Never trust `$request->input()`/`$request->all()` directly. Two patterns are equally acceptable here: inline `$request->validate([...])` at the top of the action (see `ProjectController::store()`, `KanbanBoardController`) for simple/one-off rules, or a dedicated `FormRequest` class under `app/Http/Requests/<Feature>/` (see [`SendMessageRequest`](app/Http/Requests/Chat/SendMessageRequest.php)) once the rules get long, need custom messages, or are reused. Reach for a `FormRequest` once validation grows past a handful of simple rules; otherwise inline `validate()` is fine, and there's no need to add a `FormRequest` class for a single trivial field.
- **Every form needs its own client-side checks too, not just server-side validation.** Don't rely purely on a round trip to surface backend errors through `useForm`'s `errors` object. Catch the obvious cases (empty required fields, format, min/max length) before submitting, the way `getPasswordStrength()` in [`lib/passwordStrength.ts`](resources/js/lib/passwordStrength.ts) and the `required`/`type`/`maxLength` input attributes across the auth pages already do. This is purely a UX layer for immediate feedback and fewer round trips. It doesn't replace backend validation, which is the actual security boundary and must run regardless of what the frontend already checked (see Security below).
- **Talk to the backend through Inertia's router, not `axios` or raw `fetch()`.** Use `useForm()` or `router.post/get/put/patch/delete()` from `@inertiajs/react` for application requests: it handles the CSRF token for you, gives you `errors`/`processing` state without extra plumbing, and drives Inertia's page-prop lifecycle instead of hand-rolled `.then()`/`.catch()` chains. If a UI interaction seems to need a JSON-style background request, first adapt the backend contract so it works through Inertia props, redirects, partial reloads, or Wayfinder-generated route helpers with `router.*`. Do not add new `axios` usage or new direct `fetch()` call sites. Existing legacy direct-request paths should be migrated to Inertia; once they are gone, remove `axios` from `package.json`, `package-lock.json`, and `node_modules`.
- **Declare model mass-assignment protection with the `#[Fillable]` attribute, not the `protected $fillable = [...]` property:**
  ```php
  use Illuminate\Database\Eloquent\Attributes\Fillable;

  #[Fillable([
      'card_label_color_hex',
  ])]
  class CardLabelColor extends Model
  {
      // ...
  }
  ```
  This is the convention across every model in `app/Models/`. Match it for any new model instead of reintroducing the old property form.

## Security

Every feature, frontend and backend, needs to be built defensively, not just to pass the happy path. Concretely, in this codebase that means:

- **Mass assignment**: every model declares its writable columns via `#[Fillable]` (see above). Never call `Model::unguard()` or widen a model's fillable list just to make an error go away.
- **Authorization is enforced server-side, in middleware/policies, never inferred from what the frontend shows or hides.** Project-scoped routes already do this with `project.member` / `project.role:OWNER,ADMIN` middleware ([`EnsureProjectMember`](app/Http/Middleware/EnsureProjectMember.php), [`EnsureProjectRole`](app/Http/Middleware/EnsureProjectRole.php)) and `ProjectPolicy`/`ChatRoomPolicy`. A new privileged action needs the equivalent check, not just a hidden button.
- **Validate on the backend even when the frontend already validates the same field.** Frontend validation is a UX convenience, not a security boundary (see "Always validate input in the controller" above).
- **User-controlled HTML/Markdown must go through sanitization before rendering.** The LLM chat message renderer is the reference implementation ([`LlmChatMessageList.tsx`](resources/js/components/llm-chat/LlmChatMessageList.tsx)): `rehype-sanitize` strips scripts and event handlers after `rehype-raw` parses raw HTML. Never add a new `dangerouslySetInnerHTML` or raw-HTML-from-user-input render path without going through an equivalent sanitizer.
- **File uploads are validated for type, size, and count** at the `FormRequest`/`validate()` layer. See the attachment rules in `SendMessageRequest` for reference: a max size of 51200 KB per file, and a max of 10 files. Never accept an upload without both a size and count ceiling.
- **CSRF tokens are required on legacy non-GET direct request paths until they are migrated.** Do not add new raw `fetch()` or `axios` request paths. For existing direct callers, pull the token from `meta[name="csrf-token"]` and send it as `X-CSRF-TOKEN`, matching the existing legacy pattern.
- **Never log, echo, or return secrets or credentials** (`GEMINI_API_KEY`, `REVERB_APP_SECRET`, OAuth client secrets, session tokens) in responses, error messages, or `Log::` calls.
- **Response security headers are centralized** in [`SecurityHeaders`](app/Http/Middleware/SecurityHeaders.php) (`X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`). Extend this middleware rather than setting headers ad hoc in a controller.
- Passwords are hashed via bcrypt (`BCRYPT_ROUNDS`), and two-factor auth uses TOTP (`app/Support/Totp.php`). Don't add a parallel or weaker auth path for a new feature; reuse the existing `auth`/`verified`/`two-factor` middleware stack.

### Real-time / background features

- Chat uses Laravel Reverb with `laravel-echo`/`pusher-js` on the frontend ([resources/js/echo.ts](resources/js/echo.ts)). Broadcasting channel auth is in [routes/channels.php](routes/channels.php), and `MessageSent` is the broadcast event.
- LLM chat (`app/Models/LlmChat/*`, `LlmChatController`) persists to MongoDB (`MONGODB_URI` env var, connection in `app/Services/MongoDB/MongoConnection.php`) rather than the primary MySQL database. Keep that in mind when writing queries or migrations for this feature: it won't go through Eloquent's default MySQL connection or standard migrations.
- Gemini requests run from the PHP-FPM app container through Guzzle. Keep `ext-curl` and CA certificates in the production runtime; `docker/Dockerfile` verifies the extension during the image build.
- Chat attachments and note images use `StorageService`. Local development sets `CHAT_STORAGE_DISK=public`; production sets it to `s3`. Cloudflare R2 uses the same Flysystem S3 disk and endpoint settings as other S3-compatible providers. Do not add an R2-only library or request public object ACLs. The app resolves stored relative paths into short-lived signed preview and download URLs, which allows the object bucket to stay private.
